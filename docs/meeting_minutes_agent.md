# 会议纪要整理 Agent 设计文档

## 概述

本文档描述一个**项目记忆系统**的设计方案。该系统不仅完成会议录音转写和纪要生成，更重要的是构建持续更新的"项目记忆"，包括：

- **会议纪要** - 每次会议的结构化记录
- **项目时间线** - 关键节点、决策、里程碑的时序记录
- **待办追踪** - Action Items 的全生命周期管理
- **项目上下文** - 持续更新的项目"记忆"文件
- **智能提醒** - 会议前的上下文提示，发现遗漏事项

## 配置文件格式

### 1. 项目配置 (`_project.json`)

```json
{
  "name": "MeetingEZ 2.0",
  "description": "智能会议纪要系统",
  "start_date": "2025-03-01",
  "team": ["张三", "李四", "王五"],
  "tags": ["产品", "后端", "AI"],
  "meeting_types": [
    {"type": "review", "name": "评审会", "focus": "决策"},
    {"type": "weekly", "name": "周会", "focus": "进度"},
    {"type": "brainstorm", "name": "头脑风暴", "focus": "创意"},
    {"type": "retro", "name": "复盘会", "focus": "改进"}
  ]
}
```

**字段说明：**

| 字段 | 必需 | 说明 |
|------|------|------|
| `name` | 是 | 项目名称 |
| `description` | 否 | 项目描述 |
| `start_date` | 否 | 项目启动日期 |
| `team` | 否 | 团队成员列表 |
| `tags` | 否 | 项目标签 |
| `meeting_types` | 否 | 会议类型定义 |

### 2. 人员角色映射 (`_people.json`)

```json
{
  "people": {
    "张三": {
      "role": "技术负责人",
```

- 纪要中准确标注角色

- 识别 ASR 中的人名别名

### 3. 会议元信息 (`_meeting.json`)

```json
{
  "date": "2025-03-18",
  "title": "技术方案讨论",
  "type": "review",
  "participants": ["张三", "李四", "王五"],
  "host": "张三",
  "notes": "本次会议重点关注数据库选型和微服务架构方案。请评估 PostgreSQL vs MySQL 的优劣。",
  "expected_actions": [
    "确认数据库选型",
    "确定微服务拆分方案"
  ],
  "language": "zh-CN"
}
```

**字段说明：**

| 字段 | 必需 | 说明 |
|------|------|------|
| `date` | 是 | 会议日期，格式 `YYYY-MM-DD` |
| `title` | 是 | 会议名称 |
| `type` | 否 | 会议类型（review/weekly/brainstorm/retro），影响纪要模板 |
| `participants` | 否 | 参会人员列表 |
| `host` | 否 | 主持人 |
| `notes` | 否 | 本次纪要撰写的特别要点 |
| `expected_actions` | 否 | 预期产出的行动项 |
| `language` | 否 | 会议语言，默认 `zh-CN` |

## Prompt 设计

### System Prompt

```
你是一位专业的项目管理助手，负责分析会议录音转写文本并维护项目记忆。

## 你的职责

1. **生成会议纪要** - 根据会议类型使用对应模板，客观记录讨论内容
2. **提取行动项** - 识别明确的待办事项，包含负责人和截止时间
3. **追踪完成状态** - 对比现有待办，标记已完成或提及的事项
4. **更新时间线** - 提取关键决策、里程碑、风险事件
5. **维护项目上下文** - 更新项目状态、风险、进展

## 输出要求

1. **客观准确** - 只记录会议中实际讨论的内容，不臆测
2. **结构清晰** - 使用标准格式，便于阅读和检索
3. **关联追踪** - 将新待办与现有待办关联，识别重复或延续
4. **风险敏感** - 主动识别潜在风险和阻塞项
5. **智能提醒** - 指出未被跟进的重要事项

## 会议类型识别

- **review** (评审会): 侧重决策、结论、方案对比
- **weekly** (周会): 侧重进度、计划、阻塞
- **brainstorm** (头脑风暴): 侧重创意、发散、不追求结论
- **retro** (复盘会): 侧重问题分析、改进措施
- **kickoff** (启动会): 侧重目标、分工、时间线

## 人名处理

- 使用 _people.json 中的标准名称
- 如遇到别名，自动映射到标准名称
- 如果识别到新人，在输出中标注
```

```bash
# 基本命令
python -m meeting_agent run                    # 处理所有待处理会议

python -m meeting_agent run --force            # 强制重新处理所有文件
python -m meeting_agent run --force-asr        # 强制重新 ASR
python -m meeting_agent run --force-minutes    # 强制重新生成纪要

python -m meeting_agent init-meeting "2025-03-20_周会" --date 2025-03-20 --type weekly

# 待办管理
python -m meeting_agent actions                # 列出所有待办
python -m meeting_agent actions --overdue      # 仅显示超期待办
python -m meeting_agent complete A001          # 标记待办完成
```
