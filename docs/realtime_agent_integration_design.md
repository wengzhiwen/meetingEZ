# MeetingEZ 实时会议与离线 Agent 融合方案（概要设计）

## 1. 背景

当前项目里已经有两条能力链路：

1. **实时会议 Web 端**
   浏览器采集麦克风或标签页音频，通过 OpenAI Realtime API 的 WebRTC transcription session 获取实时字幕，并对完成条目做轻量后置修正与翻译。

2. **离线会议纪要 Agent**
   扫描会议目录，使用 GLM-ASR 完成完整录音转写，再基于 GPT 生成纪要、行动项、时间线、项目上下文。

它们各自都能工作，但现在的边界也很明显：

- 实时链路的价值主要停留在“会中看字幕”和“导出文本”
- Agent 链路的价值主要停留在“会后批处理”
- 两边没有共享同一个“会议对象”
- 实时转写结果与会后高精转写结果没有版本关系
- 命令行 Agent 的主要能力还没有对应的 Web GUI

这份文档的目标，是先给出**产品与系统层面的连接思路**，不进入 API 细节。

相关术语的统一边界，见：[`docs/TERMINOLOGY.md`](/Users/wengzhiwen/dev/meetingEZ/docs/TERMINOLOGY.md)。

## 2. 核心判断

### 2.1 把“会议”变成统一资产，而不是两套工具各自的临时输入

建议把一场会议统一抽象成一个 **Meeting Workspace**，它既能被实时 Web 端写入，也能被离线 Agent 消费。

一个会议目录中不再只有“录音文件 + transcript.json”，而是同时保存：

- 会议元信息
- 原始完整录音
- 实时转写草稿
- 会后高精转写正式稿
- 纪要
- 行动项
- 时间线更新
- 项目上下文快照/提示

这样，“实时工具”和“离线 Agent”不再是两个平行产品，而是同一条会议生命周期上的两个阶段。

### 2.2 明确数据权威关系

建议采用“三层文本”的思路：

- **实时字幕**：给会中阅读和即时理解，允许不稳定
- **会后正式转写**：由完整录音重跑 GLM-ASR，作为纪要和项目记忆的唯一文本来源
- **会议纪要/行动项/时间线**：建立在正式转写之上，而不是直接建立在实时字幕之上

这正好呼应你的想法，而且我认为这是这次融合设计里最值得优先落地的一条原则。

### 2.3 本轮确认的三条硬原则

结合当前讨论，建议把下面三条直接提升为方案约束：

#### 原则 A：两条线路必须能完全独立运行

这不是“最好如此”，而是设计红线。

- 实时 Web 端可以继续作为一个轻量独立工具存在
- 离线 Agent 可以继续通过命令行独立处理录音与项目记忆
- 协同模式只是增强层，不得让任一线路对另一条线路形成运行时强依赖

换句话说：

- 没有 Agent，实时会议照常可用
- 没有 Web 端，离线整理照常可用
- 只有当用户选择协同模式时，两条线才共享会议资产并相互增强

#### 原则 B：会后整理必须反哺下次会议的实时精度

会后链路不应只是“生成一份纪要”，而应持续产出可复用的识别增强材料。

这些增强材料包括但不限于：

- 已确认术语表
- 人名 / 别名 / 职责映射
- 常见产品名、模块名、缩写
- 双语术语对照
- 最近会议中的高频议题与未完成行动项

这些内容要在下一次会议开始前，被整理成一个轻量的 **Meeting Context Pack**，反哺实时转写与实时后置处理。

#### 原则 C：语言模式必须成为会议级设定，并驱动两条线路的处理策略

不同会议不是简单地“换个 language 参数”。

至少要区分两类：

- **单主语言会议**
  例如 99% 中文，夹杂少量英语技术词汇
- **双语言会议**
  例如中文 + 日语，且两种语言都是真正的会议语言

这两类会议在实时链路和会后链路中的最佳 prompt、最佳后处理逻辑、最佳展示方式都不同。

因此，语言模式必须成为一场会议的基础画像之一，并同时驱动：

- 实时 ASR 提示策略
- 实时后置处理与展示策略
- 会后 GLM-ASR 提示策略
- 纪要生成与行动项提取策略

## 3. 设计目标

### 3.1 必须同时满足的目标

- 两套能力仍可独立运行
- 联动时能互相增强，而不是互相绑死
- 实时链路继续追求低延迟
- 会后链路继续追求高准确率和项目记忆沉淀
- 会后产出必须对下一次实时会议的识别精度产生正向作用
- 单语会议和双语会议必须有不同的处理策略
- 给现有 CLI 工具补一套足够轻量的 Web GUI
- 保持渐进式演进，避免一次性推翻现在的结构

### 3.2 本阶段刻意不做的事

- 不先设计细 API
- 不先设计复杂权限体系
- 不先做大型工作流编排平台
- 不先追求“所有事情都实时自动化”

## 4. 总体方案

## 4.1 统一对象：Meeting Workspace

建议把会议目录升级为统一工作空间。一个典型结构可以是：

```text
meetings/
└── 2026-03-22_客户周会/
    ├── _meeting.json
    ├── assets/
    │   ├── raw_full_recording.webm
    │   ├── live_segments.json
    │   ├── live_transcript.json
    │   └── bookmarks.json
    ├── transcript.json
    ├── transcript.compare.md
    ├── minutes.md
    ├── actions.md
    ├── timeline.md
    └── pre_meeting_hint.md
```

其中最关键的是：

- `raw_full_recording.*` 是会后处理的源材料
- `live_transcript.json` 只是会中草稿，不是最终依据
- `transcript.json` 是会后正式稿
- `transcript.compare.md` 用来解释“实时稿”和“正式稿”差异

## 4.2 统一生命周期：会前、会中、会后

### 会前

Agent 侧已有的项目记忆能力，反过来服务实时会议：

- 从 `context.md`、`actions.md`、近期纪要生成会前提示
- 从术语表中抽取项目关键词
- 从最近会议提炼“本次高概率出现的人名、术语、未完成事项”
- 将项目、会议类型、参会人、上下文摘要注入实时会议页面

会前的价值不是“提前生成纪要”，而是：

- 帮人快速进入状态
- 给实时转写提供术语提示
- 给会后结构化分析提供更清晰的会议元数据

这里建议形成一个抽象概念：

- **Meeting Context Pack**

它不是新的复杂系统，而是一份面向本次会议的轻量上下文集合，用来同时服务实时线和会后线。它可以包含：

- 本次会议类型
- 语言模式
- 主要参会人及别名
- 已确认术语
- 最近待办与关键决策
- 本次高概率议题

### 会中

实时端继续保持原来的核心职责，但新增“会议资产沉淀”：

- 保留完整录音
- 保存实时转写段落与时间戳
- 允许用户添加书签或轻量标记
- 记录一些会中事件，例如“这里是决策点”“这里需要 follow-up”

这里的实时文本不再背负“最终纪要原文”的责任，而是承担：

- 字幕
- 检索
- 回看定位
- 会后处理辅助信息

### 会后

结束会议后，启动一个明确的后处理阶段：

1. 使用完整录音进行 GLM-ASR 重转写
2. 生成正式转写稿
3. 对比实时稿与正式稿，提炼关键修正
4. 用正式稿生成纪要、行动项、时间线、上下文更新
5. 将分析结果反馈给项目记忆系统
6. 从本次会议中提炼可复用的识别增强信息，更新下一次会议可使用的 Context Pack

这会让实时链路和 Agent 链路形成自然分工：

- Web 负责“即时性”
- Agent 负责“准确性”和“记忆化”

## 5. 重点功能设计

## 5.1 功能一：完整录音保留 + 会后正式转写

这是第一优先级功能。

### 设计意图

- 避免直接拿实时分段结果拼纪要
- 让正式转写基于完整上下文
- 保留以后重跑模型、重做纪要的能力

### 建议规则

- 实时阶段产出 `live_transcript`
- 会后阶段基于完整录音产出 `final_transcript`
- `minutes.md` 只依赖 `final_transcript`
- UI 中明确标识“草稿字幕”和“正式转写”

### 价值

- 精度更高
- 可回溯
- 便于模型升级后重算
- 便于日后加术语纠偏、说话人修正等能力

## 5.2 功能二：实时结果与正式结果的“差异回放”

这不是锦上添花，而是连接两条链路的关键解释层。

### 建议表现

- 正式转写完成后，展示“修正摘要”
- 标注哪些专有名词被纠正
- 标注哪些行动项表述发生变化
- 支持按时间点回跳到录音位置

### 作用

- 增强用户对会后正式稿的信任
- 帮助快速审阅 AI 修正是否合理
- 为后续术语表沉淀提供候选来源

## 5.3 功能三：项目记忆反哺实时会议

这是“既能独立运行，又能协同提升”的核心。

建议把 Agent 侧已有产出反哺给实时端：

- 术语表反哺实时 ASR prompt
- 人名 / 别名映射反哺实时后置纠错
- 近期行动项反哺会前提醒
- 项目上下文反哺实时翻译/纠错 prompt
- 会议类型反哺实时侧边栏模板
- 最近几次会议中新增的双语词对反哺翻译与语言判定

例如：

- 如果这是周会，侧边栏默认展示“上次待办完成情况”
- 如果这是评审会，侧边栏默认展示“待确认决策点”
- 如果术语表里已有产品名、人名、模块名，实时识别命中率会更高

这里建议明确一个方向：

### 会后不是只产出“文档”，还产出“识别资产”

这些识别资产可以分层沉淀：

- **长期资产**
  术语、人名、别名、固定翻译、项目模块名
- **短期资产**
  最近 3 到 5 次会议的高频主题、近期待办、最近决策
- **本次会议资产**
  本场会议特有的议程词、客户名、版本号、临时代号

对实时链路来说，真正有价值的是“把这三层资产组合成 prompt 与后处理线索”，而不是简单地把旧纪要全文塞进去。

## 5.4 功能四：实时侧的书签/标记回流到会后分析

实时会议中很适合让用户做极轻量的人工增强，而不是要求人工记完整笔记。

可支持的标记类型：

- 决策点
- 待办候选
- 风险点
- 需要复听
- 精彩片段

这些标记在会后可作为 Agent 的分析引导材料：

- 提高纪要结构化质量
- 降低长会议信息淹没
- 让用户真正觉得“会中操作会改善会后结果”

## 5.5 功能五：命令行工具的简单 Web GUI

建议不是再做一个新产品，而是在现有 Flask Web 上补一层“项目工作台”。

目标不是替代 CLI 的所有能力，而是让高频操作更可见、更可点。

### 最小可用 GUI 组成

#### 1. 项目总览页

展示：

- 项目列表
- 会议总数
- 待处理会议数
- 待办总数 / 超期待办
- 最近一次处理时间

#### 2. 会议列表页

展示每场会议的状态：

- 是否已有完整录音
- 是否已有实时稿
- 是否已有正式转写
- 是否已生成纪要
- 是否存在待审核术语 / 待解释问题

支持操作：

- 新建会议
- 上传录音
- 启动后处理
- 查看详情

#### 3. 会议详情页

展示一场会议的完整生命周期：

- 会议元信息
- 录音资产
- 实时稿
- 正式稿
- 差异摘要
- 纪要
- 行动项
- 时间线影响

#### 4. 待办与项目记忆页

展示：

- `actions.md` 中的行动项
- `timeline.md` 中的关键节点
- `context.md` 中的项目背景摘要

支持基础操作：

- 标记待办完成
- 查看超期项
- 跳回来源会议

#### 5. 术语与待解释问题页

这是对当前 Agent 非常有帮助的一个轻量 GUI：

- 查看待确认术语
- 确认 / 拒绝术语建议
- 查看 `_context.md` 里待人工补充的问题

这类页面虽然简单，但会明显提升长期可维护性。

## 5.6 功能六：语言模式驱动的双管线策略

这是你第三点里最值得单独成章的一项。

建议不要把语言处理理解为“用户随手选一个 primary/secondary language”，而是把它提升为会议画像的一部分。

### 模式一：单主语言会议

典型场景：

- 绝大多数内容是中文
- 会出现少量英文技术词、缩写、产品名
- 用户核心诉求是稳定、低延迟、少误翻译

这类会议的建议策略是：

- 实时线使用**单主语言优先**的提示策略
- 明确告诉模型“主要语言是中文，但要保留英语技术词原样”
- 默认不做逐段双语翻译
- 后置处理重点放在术语保留、错别字纠正、人名修正
- 会后正式转写也使用相同语言画像，避免把技术词误归一成中文近音词

这类会议最怕的不是“漏翻译”，而是：

- 技术词被错误汉化
- 缩写被听错
- 人名、模块名被误替换

因此单主语言会议的优化重点是**术语保真**。

### 模式二：双语言会议

典型场景：

- 中文和日语都会被持续使用
- 同一场会议中可能频繁切换语言
- 用户需要保留原文，同时帮助另一方理解

这类会议的建议策略是：

- 实时线启用**语言感知型处理**
- 每段保留原始语言
- 按需追加主语言翻译
- 术语表中维护双语词对和人名多语言写法
- 会后正式转写需要保留段级语言信息，避免强制归并成单语文本
- 纪要生成时优先基于正式稿做语义整合，但可保留关键原文引用

这类会议最重要的是**语言边界清晰**：

- 什么是原文
- 什么是翻译
- 哪些词应该保留原语言
- 哪些句子可以归并成主语言摘要

### 会议语言模式如何同时作用于两条线

#### 对实时线的作用

- 决定实时 ASR prompt 风格
- 决定是否默认开启翻译
- 决定 UI 展示是“单文本流”还是“原文 + 翻译”
- 决定术语纠错偏向“保留英文术语”还是“维护双语映射”

#### 对会后线的作用

- 决定 GLM-ASR 的提示方式
- 决定转写结果是否要保留语言标签
- 决定纪要 prompt 更偏“单语总结”还是“跨语义整合”
- 决定行动项抽取时是否需要参考双语表达对齐

因此，更准确的设计应该是：

- **会议类型** 决定纪要结构重点
- **语言模式** 决定 ASR、后处理、展示和总结策略

两者共同构成会议画像。

## 6. “独立工作”和“协同工作”的边界设计

这是这次设计里必须守住的原则。

### 6.1 实时 Web 端独立运行

即使不接 Agent，也应继续可用：

- 直接开始会议
- 看实时字幕
- 下载文本
- 保存本地记录

也就是说，现有“轻量直连模式”不能被破坏。

进一步说：

- 不应要求用户先初始化项目才能开会
- 不应要求服务端 Agent 在线，实时页才能工作
- 不应让“协同模式”拖慢“轻量模式”

### 6.2 Agent 独立运行

即使不用 Web 端采集，也应继续可用：

- 手动创建会议目录
- 手动放入录音文件
- 命令行跑 ASR / 纪要 / 项目记忆

进一步说：

- Agent 不应依赖浏览器生成的实时草稿才能生成正式稿
- Web 侧资产如果缺失，Agent 仍然能完成核心工作

### 6.3 协同模式

只有当用户明确选择“项目化会议”时，两者才进入协同：

- Web 端写入 Meeting Workspace
- Agent 端消费同一个 Workspace
- Web GUI 展示 Agent 的处理结果

也就是说，协同是增强层，不是前置依赖。

## 7. 推荐的产品形态

我建议把产品形态从“一个实时字幕页 + 一个 CLI 工具”升级为“双模式产品”：

### 模式 A：即时会议模式

适合临时会议、快速转写、轻量使用。

特点：

- 低门槛
- 浏览器即开即用
- 不要求项目归档

### 模式 B：项目会议模式

适合长期项目、需要沉淀行动项与项目记忆的会议。

特点：

- 先选择项目 / 会议类型 / 参会人
- 自动挂接项目上下文
- 会后进入正式处理
- 结果沉淀进项目记忆

这样的双模式能兼顾“轻”和“深”。

## 8. 实现策略建议

## 8.1 第一阶段：先把链路接上

优先做最有杠杆的连接，不求一步到位。

建议先落地：

- 完整录音保存
- 会后自动触发 GLM-ASR 正式转写
- 正式转写替代实时稿作为纪要输入
- 增加会议列表/详情的简单 Web GUI

这一阶段完成后，系统就已经从“两套独立工具”升级成“一条完整会议链路”。

## 8.2 第二阶段：做协同增强

再补：

- 术语表反哺实时识别
- 会前提示进入 Web
- 书签/标记回流会后分析
- 正式稿与实时稿差异展示

## 8.3 第三阶段：做项目工作台

最后再补：

- Actions / Timeline / Context 的 Web 管理页
- 术语审核页
- 待解释问题处理页

## 9. 风险与权衡

### 9.1 浏览器长时间完整录音的稳定性

如果会议很长，浏览器内存和录音上传方式需要谨慎设计。

产品上要接受一个现实：

- “实时分段上传”适合低延迟
- “完整录音保留”适合会后处理

这两者是并行链路，不应强行合并成同一套传输模型。

### 9.2 数据存储量会上升

保存完整录音后，存储成本会明显增加。

所以建议尽早定义：

- 原始录音保存多久
- 是否允许手动归档/删除
- 是否区分草稿资产与正式资产

### 9.3 用户认知需要清晰

必须在界面上明确告诉用户：

- 会中看到的是“实时草稿”
- 会后会得到“正式稿”
- 纪要基于“正式稿”生成

只要这个认知不清，用户就会混淆为什么两份文本不完全一样。

## 10. 我最推荐的落地顺序

如果只选一条主线，我建议按下面顺序推进：

1. **统一会议工作空间**
   让实时端和 Agent 至少共享同一份会议资产目录。

2. **引入会议画像**
   至少把“会议类型 + 语言模式 + 参会人 / 项目上下文”变成统一设定。

3. **完整录音保留**
   把完整录音变成会后处理的标准输入。

4. **会后 GLM 正式转写**
   明确正式稿是纪要唯一输入来源。

5. **识别资产回流**
   把术语、人名、双语词对、近期议题沉淀成下一次可复用的 Context Pack。

6. **最小 Web 工作台**
   先做项目页、会议列表页、会议详情页。

7. **项目记忆反哺实时会议**
   再把术语表、会前提示、待办上下文反向接入实时页。

## 11. 结论

这次融合不应该理解为“给实时页再加几个按钮”，而应该理解为：

**把实时会议能力和离线项目记忆能力，整合成一条完整的会议生命周期。**

其中最重要的产品原则有三条：

- 实时字幕负责快，正式转写负责准
- 两条线路必须能完全独立运行，协同只是增强层
- 会议资产统一沉淀，避免两边各自产生孤岛数据
- 会后整理必须反哺下次会议的实时精度
- 单语会议和双语会议必须按不同画像选择不同处理链路
- Web GUI 先覆盖高频查看与触发动作，不急着覆盖全部 CLI 能力

如果沿着这个方向推进，MeetingEZ 会从“一个很好用的实时字幕工具 + 一个很强的离线 Agent”，变成“一个既能即时辅助会议、又能持续积累项目记忆的会议操作系统”。
