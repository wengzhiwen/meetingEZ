# MeetingEZ 术语约定与命名边界

## 目的

当前仓库同时承载两条能力线：

- 实时会议 Web 端
- 离线会议后处理 Agent

随着方案往“独立运行 + 协同增强”演进，项目里已经出现一些容易混淆的术语：

- `转写` / `转录`
- `live transcript` / `final transcript` / `正式转写`
- `术语表` / `专有名词表` / `glossary`
- `context.md` / `_context.md`
- `行动项` / `待办`

这份文档的目标，是给后续设计、文档、UI 文案、代码命名提供一套统一参照，减少讨论和实现过程中的歧义。

## 一、总原则

### 1. 中文主名优先，英文只作为补充别名

后续文档里优先使用中文主名，必要时首次出现补一个英文别名：

- 会议画像（Meeting Profile）
- 会议工作空间（Meeting Workspace）
- 术语表（Glossary）

不要在同一层概念里频繁来回切换中英文主叫法。

### 2. 区分“界面展示对象”和“持久化资产”

很多混乱都来自把 UI 中显示的字幕，和系统真正保存的会议资产混成一个概念。

后续统一按下面思路区分：

- `字幕`：偏界面展示
- `转写`：偏语音转文字结果
- `纪要`：偏总结与结构化产出
- `项目记忆`：偏跨会议持续沉淀的资产

### 3. 不单独使用 `final transcript` 这个词

这是当前最容易产生歧义的一组词。

原因是：

- 在 Realtime API 语境里，`final` 往往只是指某个实时条目已经完成
- 在会议后处理语境里，`final` 容易又被理解成“会后正式转写稿”

因此，后续应避免脱离上下文单独说 `final transcript`，必须带限定词。

## 二、推荐的统一术语

## 2.1 产品与子系统

| 层级 | 推荐术语 | 英文别名 | 说明 |
|------|----------|----------|------|
| 产品整体 | MeetingEZ | MeetingEZ | 整个项目/产品名 |
| 实时子系统 | 实时会议 Web 端 | Realtime Meeting Web App | 浏览器内进行实时识别与展示的部分 |
| 离线子系统 | 会议后处理 Agent | Meeting Agent / Post-Meeting Agent | 命令行侧的录音处理、纪要生成、项目记忆维护 |
| 能力集合 | 项目记忆系统 | Project Memory System | `context.md` / `timeline.md` / `actions.md` / 术语与背景知识等的总称 |

### 约定

- `Agent` 指离线后处理能力，不指实时 Web 端
- `项目记忆系统` 是能力层概念，不等同于某个单独命令

## 2.2 会议对象

| 推荐术语 | 英文别名 | 说明 |
|----------|----------|------|
| 会议元信息 | Meeting Metadata | `_meeting.json` 中的基础信息，如日期、标题、类型、参会人 |
| 会议画像 | Meeting Profile | 在元信息基础上，增加语言模式、上下文、术语偏好等后的完整处理画像 |
| 会议目录 | Meeting Directory | 当前文件系统中的会议文件夹 |
| 会议工作空间 | Meeting Workspace | 面向未来协同设计的语义概念；通常由会议目录承载 |

### 约定

- 现在落地实现仍以“会议目录”为主
- 在设计文档中可以用“会议工作空间”表达未来统一资产概念
- “会议画像”比“会议元信息”多一层决策含义，不能混用

## 2.3 音频资产

| 推荐术语 | 英文别名 | 说明 |
|----------|----------|------|
| 音频输入源 | Audio Source | 麦克风或标签页音频 |
| 实时音频流 | Live Audio Stream | Realtime 会话中的实时音频输入 |
| 会议录音 | Meeting Recording | 一场会议产生的录音文件 |
| 原始完整录音 | Raw Full Recording | 会后正式处理的源音频，强调完整、可回放、可重跑 |
| 分片音频 | Audio Chunk / Segment | ASR 分块处理时的中间音频切片 |

### 约定

- 讨论实时链路时优先说“实时音频流”
- 讨论会后重跑时优先说“原始完整录音”
- 不建议把浏览器中的实时流直接叫“录音文件”

## 2.4 文本资产

这是本项目最需要统一的一组概念。

| 推荐术语 | 英文别名 | 说明 |
|----------|----------|------|
| 增量字幕 | Live Transcript Delta | 实时识别过程中尚未完成的增量文本 |
| 实时定稿条目 | Realtime Completed Item | Realtime API 某个 `item_id` 完成后的条目文本 |
| 实时字幕流 | Realtime Transcript Stream | 会中 UI 所见的整条实时字幕流，包含多个实时定稿条目 |
| 原始转写 | Raw Transcript | ASR 直接返回、未经过修正的文本 |
| 修正后转写 | Corrected Transcript | 经术语/上下文修正后的文本 |
| 正式转写稿 | Official Transcript | 会后基于完整录音生成、用于纪要和项目记忆的权威文本 |
| 转写片段 | Transcript Segment | 带时间戳的持久化转写单元，如 `TranscriptSegment` |
| 差异对照稿 | Transcript Diff / Compare View | 实时结果与正式转写稿之间的比对结果 |

### 关键边界

#### 1. “字幕”不等于“正式转写稿”

- 字幕是面向会中阅读体验的展示对象
- 正式转写稿是面向会后纪要与项目记忆的权威文本资产

#### 2. “实时定稿条目”不等于“正式转写稿”

Realtime API 中某个条目 completed，只能说明它在会中状态上已完成，不能等同于会后权威稿。

#### 3. “原始转写”和“修正后转写”是后置处理层概念

这两个词更适合描述：

- 当前实时 Web 端里 `rawTranscript`
- 当前实时 Web 端里 `correctedTranscript`

它们仍属于会中链路，不应直接替代“正式转写稿”。

## 2.5 总结与记忆资产

| 推荐术语 | 英文别名 | 说明 |
|----------|----------|------|
| 会议纪要 | Meeting Minutes | `minutes.md`，本次会议的结构化总结 |
| 行动项 | Action Item | 从会议中提炼出的可追踪任务 |
| 待办 | Todo / Task List Item | 面向列表和 UI 的简化叫法，可视为行动项的展示别名 |
| 项目时间线 | Project Timeline | `timeline.md`，跨会议的里程碑、决策与风险演进 |
| 项目上下文 | Project Context | `context.md`，自动维护的项目状态摘要 |
| 会议前提示 | Pre-Meeting Hint | `pre_meeting_hint.md`，面向下一次会议的提示信息 |
| 项目记忆 | Project Memory | 上述多份资产的集合概念 |

### 约定

- 设计和数据模型层优先使用“行动项（Action Item）”
- UI 和列表标题可以继续用“待办”这种更短的词
- “项目记忆”是总称，“项目上下文”只是其中一个文件

## 2.6 术语与背景知识资产

当前项目里这一块最需要合理区分。

| 推荐术语 | 文件/对象 | 说明 |
|----------|-----------|------|
| 术语表 | `_glossary.json` | 已确认术语及其别名、类型 |
| 待审核术语 | `_glossary_pending.json` | LLM 提议、待确认的术语 |
| 术语拒绝表 | `_glossary_rejected.json` | 已明确拒绝的术语建议 |
| 项目背景说明 | `_context.md` | 人工维护的解释性背景知识 |
| 组合上下文 | `get_combined_context()` | 术语表 + 项目背景说明拼接出的 prompt 参考 |

### 推荐统一叫法

- `glossary` 统一对应“术语表”
- `_context.md` 不再笼统叫“context”，建议统一叫“项目背景说明”或“背景知识说明”

### 为什么要单独区分 `_context.md`

因为它和 `context.md` 的语义完全不同：

- `context.md` 是自动维护的项目状态摘要
- `_context.md` 是人工维护的背景知识、概念解释和补充说明

如果两者都叫“上下文”，后续提示词设计、GUI 页面设计、代码重构都会非常混乱。

## 三、当前项目里最值得统一的几组概念

## 3.1 `转写` vs `转录`

### 推荐结论

后续统一优先使用 **转写**。

### 原因

- 当前代码和数据模型里核心对象已经是 `transcript`
- 会议语音转文字在产品语境里，“转写”比“转录”更贴近当前使用习惯
- `转录` 在一些文档中仍可作为自然语言表达存在，但不建议作为规范名

### 建议

- 新文档、新界面、新变量命名，优先使用“转写”
- 历史文档中的“转录”可逐步迁移，不必一次性全面清洗

## 3.2 `live/final` 的双重含义

### 当前风险

在实时链路中：

- `live` = 增量字幕
- `final` = 单个实时条目完成

在协同设计中：

- “正式稿”又容易被英文脑补成 `final transcript`

### 推荐结论

统一拆成三层：

- 增量字幕
- 实时定稿条目
- 正式转写稿

以后不要直接说“final transcript”，除非明确补足限定语。

## 3.3 `专有名词表` vs `术语表`

### 当前现状

- UI 中存在“专有名词表”
- 输入框又叫“会议术语表”
- Agent 里整体使用 `glossary`

### 推荐结论

统一主名为 **术语表**。

“专有名词”只是术语表中的一个子类，不应成为总称。

### 建议表述

- 产品/设计/代码层：术语表
- UI 辅助说明：术语表（含专有名词、人名、缩写、产品名）

## 3.4 `context.md` vs `_context.md`

### 当前现状

项目里实际上有两类完全不同的“context”：

#### `context.md`

- 自动生成/更新
- 记录项目状态、风险、近期关注
- 更接近“项目状态上下文”

#### `_context.md`

- 人工维护
- 记录术语解释、概念说明、背景知识
- 更接近“背景知识说明”

### 推荐结论

以后文档和讨论中统一这样叫：

- `context.md` = 项目上下文
- `_context.md` = 项目背景说明

如果要做代码层重构，建议进一步改名：

- `meeting_agent.memory.context.ContextManager`
  建议未来改为 `ProjectContextManager`
- `meeting_agent.glossary.context_manager.ContextManager`
  建议未来改为 `BackgroundContextManager` 或 `ReferenceContextManager`

这不是要求现在立刻改代码，而是先把语义边界定清楚。

## 3.5 `行动项` vs `待办`

### 推荐结论

统一采用：

- 规范术语：行动项（Action Item）
- UI 简称：待办

### 原因

- 数据模型、LLM 输出、`actions.md` 的语义更接近“行动项”
- 用户界面里“待办”更短、更易读

因此，二者不是冲突关系，而是“规范名 + 展示名”的关系。

## 四、语言相关术语

考虑到后续会显式支持“单主语言会议”和“双语言会议”，语言概念也应统一。

| 推荐术语 | 说明 |
|----------|------|
| 语言模式 | 一场会议的语言处理模式，如单主语言、双语言 |
| 主要语言 | 会议中占主导的语言 |
| 第二语言 | 需要协同处理的另一种会议语言 |
| 原始语言 | 某一条实时条目实际识别到的语言 |
| 主语言译文 | 面向主要语言读者的译文 |
| 双语词对 | 在双语言会议中应长期沉淀的术语映射 |

### 推荐区分

#### 单主语言会议

例子：

- 中文为主，夹少量英语技术词

重点是：

- 术语保真
- 人名修正
- 少误翻译

#### 双语言会议

例子：

- 中文 + 日语

重点是：

- 原文与译文边界清晰
- 双语术语映射
- 分段语言识别与跨语义整合

## 五、建议后续在文档和代码里遵守的说法

### 推荐说法

- 实时会议 Web 端
- 会议后处理 Agent
- 会议画像
- 会议工作空间
- 增量字幕
- 实时定稿条目
- 正式转写稿
- 术语表
- 项目上下文
- 项目背景说明
- 行动项
- 项目记忆

### 不推荐的模糊说法

- final transcript
- context（不加限定）
- 专有名词表（作为总称）
- 转录结果（作为规范术语）
- 字幕文件（指代所有文本产出）

## 六、对当前代码结构的命名建议

这里先给方向，不要求本轮立即改动。

| 当前概念/对象 | 当前名字 | 建议的语义名字 |
|---------------|----------|----------------|
| 自动维护的项目摘要 | `context.md` / `meeting_agent.memory.context.ContextManager` | Project Context / ProjectContextManager |
| 人工背景知识说明 | `_context.md` / `meeting_agent.glossary.context_manager.ContextManager` | Background Context / BackgroundContextManager |
| 实时 completed 文本 | `final` | Realtime Completed Item / 实时定稿条目 |
| 会后权威转写 | `transcript.json` | Official Transcript / 正式转写稿 |
| 实时术语输入 | `glossaryInput` | Meeting Glossary Input / 会议术语输入 |

## 七、对后续协同设计最重要的几条落地结论

1. “实时字幕”和“正式转写稿”必须严格分开。

2. “术语表”和“项目背景说明”都属于识别增强资产，但不是同一类资产。

3. “项目上下文”与“项目记忆”不是同义词。

4. “会议元信息”不足以驱动后续双管线，未来要补出“会议画像”。

5. 实时链路中的 `live/final` 只描述会中状态，不描述会后权威级别。

6. 后续所有新的页面、文档、字段命名，建议以本文件为统一参照。
